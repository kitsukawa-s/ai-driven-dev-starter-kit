# docs/context（補助コンテキスト）

このディレクトリは、会議メモ、チャット補足、過去の判断、未決事項、却下案、注意事項などを横断的に集める場所です。

レビュー・バグ調査・任意調査のときに、AIが「正式資料とのズレや矛盾、反映漏れ、未決事項・却下案の混入」に気づくための**確認トリガー**として参照します。

---

## 重要：docs/context/ は確定仕様ではありません

- `docs/context/` は確定仕様ではありません。
- AIは `docs/context/` の内容を、そのまま仕様・設計・実装方針として採用してはいけません。
- 正式資料（`docs/<command_or_app_name>/features/<feature_name>/20_spec.md`、`21_design.md`、`common_design/` など）が常に基準です。
- `docs/context/` は、その正式資料とのズレ・矛盾・反映漏れ・未決事項・却下案の混入を検出するための補助資料です。

正式資料と `docs/context/` が矛盾する場合、AIは勝手にどちらかを採用しません。人間確認事項として報告します。

---

## 役割

`docs/context/` に集めるものの例:

- 会議メモ
- チャットでの補足
- 過去の判断とその背景
- まだ決まっていない未決事項
- 一度検討して見送った却下案
- 実装・運用上の注意事項

これらは「決まったこと」と「決まっていないこと」「やらないと決めたこと」が混在します。
そのため、ここにある内容を確定仕様として扱ってはいけません。

---

## 使い方

### レビュー時

- 正式資料を基準にレビューします。
- `docs/context/` は補助資料として参照し、正式資料との間にズレ・矛盾・反映漏れがないかを確認します。
- ズレや矛盾を見つけても、AIは勝手に正式資料や実装を直しません。人間確認事項として記録します。

### バグ調査・バグ修正計画時

- バグ報告が、`docs/context/` の未決事項・却下案・保留事項・未反映の会議メモに基づいていないかを確認します。
- 基づいている疑いがある場合は、バグではなく仕様変更候補または人間確認事項として扱い、確定仕様として扱いません。

### 任意調査時（人間からの依頼）

- 人間は任意のタイミングで、「この機能に関係しそうな過去判断・未決事項・会議メモを `docs/context/` から探して」とAIに依頼できます。
- 依頼を受けたAIは、`docs/context/` から関係しそうな記録を探し、**確認トリガー**として提示します。
- 提示する際は、それぞれが「確定なのか/未決なのか/却下なのか」を区別し、確定仕様としては扱いません。
- AIが内容の確定可否を判断できない場合は、人間確認事項として報告します。

---

## AIの基本ルール

- `docs/context/` の内容を、そのまま確定仕様・設計・実装方針として採用しない。
- 正式資料と矛盾する場合は、勝手に解決せず人間確認事項として報告する。
- AIが判断できない内容は、確定仕様とせず人間確認事項として扱う。
- AIは、人間の明示指示なしに `docs/context/` を作成・更新しない（ここは人間が管理する入力資料です）。

---

## メモの書き方

個々のメモを追加する場合は、`docs/templates/context_note_template.md` を参照してください。
種別（会議メモ／未決事項／却下案など）と確定状況（未決／保留／却下／参考）を明示すると、確認トリガーとしての判定が安定します。

ファイル名は内容が分かる名前にしてください（例: `2026-06-04_kickoff_meeting.md`、`undecided_output_format.md`）。
