# 標準プロンプトとチャット指示の役割分担メモ

## 目的

AI駆動開発スターターキット13の改善検討として、`cli_text_counter` 作成時にチャット側で長く補足していた内容を振り返り、標準 `prompts/*.md` に入れるべき共通ルールと、毎回チャットで渡すべき作業固有情報を整理する。

今回は分析メモのみを作成する。`prompts/*.md`、`AGENTS.md`、`README.md`、`docs/templates/`、`src/`、`tests/`、既存の command/app は変更しない。

## 参照したもの

- `AGENTS.md`
- `README.md`
- `docs/how_to_use_prompts.md`
- `docs/tutorials/010_simple_calculator.md`
- `prompts/*.md`
- `docs/cli_text_counter/`
- `docs/cli_text_counter/features/text_counter/`

## 1. 標準 prompts/*.md に入れるべき共通ルール

標準プロンプトには、作業対象が変わっても毎回同じ判断になるルールを入れる。

- 参照順序: `AGENTS.md`、対象ドキュメント、必要なテンプレートを先に読む
- 作業範囲: プロンプトごとに作成または更新してよいファイルを限定する
- 禁止事項: `prompts/*.md`、`AGENTS.md`、`docs/templates/`、`src/common/`、指定外ファイルを変更しない
- 段階分離: 仕様、設計、呼び出し定義、テスト計画、レビュー観点、実装、レビューを勝手に先へ進めない
- 責務分担: feature は機能固有ロジック、entrypoint はCLI引数・feature呼び出し・標準出力・終了コードに限定する
- テスト範囲: feature 単体テスト、entrypoint テスト、結合試験の役割を分ける
- 結合試験: feature 内部の詳細ロジックを過剰に再確認しない
- レビュー結果: `05_review_checklist.md` ではなく `06_review_result.md`、command/app は `11_command_review_result.md` に記録する
- 再レビュー: 既存レビュー結果をそのまま採用せず、現在のファイル群を読み直す
- 共通化: 候補があっても `src/common/` へ勝手に切り出さず、提案にとどめる
- 標準命名: `test_entrypoint_<short_name>.py`、`test_integration_<short_name>.py` の命名ルール
- 出力先固定: 利用者が指定した出力先パスをAI判断で変更しない
- import mismatch やファイル名衝突時: 別名ファイルを勝手に作らず、確認事項として報告する

## 2. 毎回チャットで渡すべき作業固有情報

チャット側には、対象 command/app や feature によって変わる情報を残す。

- 参照するプロンプトのパス
- コマンド/アプリ名
- 対象 feature 名
- 対象機能フォルダ
- 作りたいもの
- 実行イメージまたは利用イメージ
- 今回やらないこと
- 対象 overview
- 対象 entrypoint
- 実装ファイル
- feature 単体テストファイル
- entrypoint テストファイル
- 結合試験計画ファイル
- 結合試験ファイル
- レビュー結果ファイル
- `short_name`
- 今回だけの補足条件
- レビューだけ行う、実装だけ行う、テストだけ行うなどの作業モード

特に「入力」「出力」「エラー扱い」「対象外」は仕様の芯なので、標準プロンプトに固定せず、チャットまたは対象ドキュメントから明確に渡す。

## 3. 人間が判断すべき情報

AIが仮定して進めると体験や仕様がぶれやすいものは、人間が判断する。

- command/app の目的と利用者
- feature 分割の妥当性
- 何を今回やらないことに含めるか
- エラー時の期待動作
- 空文字列、マルチバイト文字、境界値などの仕様上の扱い
- `short_name` が feature 名でよいか、command/app 名に寄せるか
- import mismatch やテストファイル名衝突が起きた場合の解消方針
- fallback import を標準方針として採用するか
- `src/common/` への共通化を認めるか
- 結合試験を作る必要があるか
- 既存ドキュメントと現在の実装状態がずれたとき、どちらを正とするか
- 標準プロンプトをどこまで厚くするか

## 4. チャット側から標準プロンプト側に寄せられそうなもの

`cli_text_counter` 作成時に長く補足していた内容のうち、汎用性が高いものは標準プロンプト側へ寄せられる。

- 指定した出力先パスをAI判断で変更しない
- `pytest` の import mismatch やファイル名衝突時に、勝手に別名ファイルを作らない
- 衝突時は、発生した問題、衝突ファイル、指定パス、標準命名との差分、代替案、推奨案、人間判断が必要な点を報告する
- entrypoint テストと結合試験で同じ観点を過剰に重ねない
- 結合試験は subprocess などで command/app としての接続を確認するが、feature 単体ロジックは再検証しすぎない
- レビュー時は古い `06_review_result.md` や `11_command_review_result.md` の判定をそのまま採用しない
- レビュー依頼だけの場合、実装ファイルやテストファイルを変更しない
- feature 単体レビューでは command/app 全体の最終判定をしない
- command/app 全体レビューでは feature 単体レビュー結果も読む
- `tasks.md` には現在地と次の一手だけを書き、仕様やレビュー詳細の代わりにしない
- `src/common/` は保護対象であり、共通化候補は提案にとどめる

一部はすでに現在の `prompts/*.md` に反映されている。今後は重複してチャットで長く書かなくてもよい候補として扱える。

## 5. 標準プロンプトに入れず、チャット側に残すべきもの

標準プロンプトに入れると汎用性を失う情報は、チャット側に残す。

- `cli_text_counter` や `text_counter` などの具体名
- `count_characters` などの関数名候補
- `--text` などのCLI引数名
- `python src/cli_text_counter/entrypoint.py --text "hello"` などの実行例
- 空文字列を許可するかどうか
- 文字数を Python の `len()` 相当で数えるかどうか
- 単語数、行数、ファイル入力など、その機能固有の対象外
- 日本語文字列をテスト観点に含めるかどうか
- subprocess の `WinError 6` のような特定環境での実績
- 今回はドキュメントだけ、今回はレビューだけ、今回は実装しない、などの一時的な制約
- 既存ファイルの現在状態に合わせた個別の更新方針

これらは標準化すると、別の command/app でも同じ仕様だとAIが誤解するリスクがある。

## 6. 今後プロンプトを短くするための改善案

チャット指示を短くするには、標準プロンプト側と利用ドキュメント側で「毎回の指定項目」を型として見せるとよい。

- `docs/how_to_use_prompts.md` に、各プロンプト用の短い依頼テンプレートを置く
- チャット例では、必須項目と任意項目を分ける
- `補足条件` に書きがちな共通ルールは、標準プロンプトに移したうえでチャット例から削る
- `short_name` の決め方を、標準ルールとしてさらに目立つ場所にまとめる
- import mismatch やファイル名衝突時の対応を、entrypoint と結合試験の共通ルールとして整理する
- レビュー専用プロンプトでは「レビューだけ行う場合は変更しない」を標準化し、チャット側では「レビュー結果ファイル」だけを指定する
- 実装系プロンプトでは、実装前に提示する確認内容を共通化し、チャット側で毎回書かない
- チュートリアルでは長い補足を減らし、「標準プロンプト側に含まれるため省略」と明記する
- 作業固有情報を、1つの入力ブロックとしてコピペしやすい形にする

例:

```text
prompts/implement_feature.md を参照してください。

対象機能フォルダ: docs/<command_or_app_name>/features/<feature_name>/
コマンド/アプリ名: <command_or_app_name>
対象機能名: <feature_name>
実装ファイル: src/<command_or_app_name>/features/<feature_name>.py
テストファイル: tests/<command_or_app_name>/features/test_<feature_name>.py
作りたいもの: ...
補足条件: なし
```

## 7. 標準プロンプトを重くしすぎるリスク

共通ルールを標準プロンプトに寄せるほど、毎回のチャットは短くなる。一方で、標準プロンプトが重くなりすぎると次のリスクがある。

- AIが読むべきルールが多くなり、重要な作業固有情報が埋もれる
- プロンプトごとの差分が見えにくくなり、どの段階の作業か判断しにくくなる
- 標準ルールが強すぎて、小さな例外や人間の意図を扱いにくくなる
- 初学者が「何を指定すればよいか」を把握しにくくなる
- 同じ注意書きが `AGENTS.md`、`docs/how_to_use_prompts.md`、`prompts/*.md`、チュートリアルに重複し、更新漏れが起きやすくなる
- 標準プロンプトに具体例を入れすぎると、別の command/app でもその具体例を仕様のように扱う可能性がある
- チャット側で人間が判断すべき情報まで、AIが標準ルールとして処理してしまう

標準プロンプトには「変わらない作業ルール」を置き、チャットには「今回の対象と判断材料」を置く、という境界を維持するのがよい。

## まとめ

標準プロンプトに寄せるべきものは、作業範囲、禁止事項、責務分担、出力先固定、命名ルール、レビュー結果の扱い、衝突時の停止ルールである。

チャット側に残すべきものは、対象 command/app、feature、具体的な入出力、今回やらないこと、出力先パス、作業モード、人間が判断した仕様である。

`cli_text_counter` の経験から見ると、長いチャット補足の多くはすでに標準プロンプトへ移せる性質を持っている。ただし、標準プロンプトを厚くしすぎると、初学者向けの見通しと作業固有の判断が弱くなるため、共通ルールと具体仕様の境界を保つ必要がある。
