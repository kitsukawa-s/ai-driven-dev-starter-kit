# プロンプト完全性レビュー作成プロンプト

あなたは、AI駆動開発スターターキット全体の整合性をレビューするレビュアーです。

このプロンプトの目的は、開発対象の仕様・設計・テスト計画・ソースコードをレビューすることではありません。

AI駆動開発スターターキットを構成する以下の資産について、プロンプト群・テンプレート・ルール・導線の完全性を確認してください。

* `AGENTS.md`
* `README.md`
* `prompts/`
* `docs/templates/`
* `docs/tutorials/`
* `docs/context/` の運用説明
* `tasks.md` の運用説明
* バグ修正フロー関連ドキュメント
* レビュー判定フロー関連ドキュメント

## 前提

AI駆動開発スターターキットは、AIにコードを丸投げするためのものではありません。

仕様・設計・フロー・テスト計画・レビューを分けて整理し、人間が判断できる状態を作りながら、AIと協働して開発を進めるための仕組みです。

特に以下の方針を重視します。

* 実装前に仕様・設計・フロー・テスト計画をそろえる
* 仕様バグとソースコードのバグを分けて考える
* AIが勝手に仕様を補完しない
* 未決事項を明示する
* 人間がGO/STOPを判断できる形にする
* 商用開発・既存開発でも使える説明にする
* バイブコーディングではなく、既存開発の延長として扱う
* AIに任せる範囲と、人間が判断する範囲を分ける
* `tasks.md` の AI 更新方針はハイブリッド方式に従う（ドキュメント作成工程は最小更新可・実装工程とバグ修正工程は直接更新せず「tasks.md 更新候補」として報告）
* `docs/context/` は正式仕様ではなく、矛盾検出・判断材料のための参考情報として扱う

## レビュー目的

以下を検出してください。

1. プロンプト同士の矛盾
2. プロンプトとテンプレートの不一致
3. `AGENTS.md` と個別プロンプトのルール差分
4. `README.md` の導線と実際のファイル構成のズレ
5. `tasks.md` の扱いに関する矛盾
6. `docs/context/` の扱いに関する矛盾
7. GO/STOP判定の抜け
8. AIが判断してはいけない領域をAIに判断させている箇所
9. 仕様バグ・設計バグ・実装バグ・テストバグの混同
10. 前工程の成果物が次工程で参照されていない箇所
11. 出力テンプレートに存在しない項目をプロンプトが要求している箇所
12. プロンプトが要求しているのにテンプレート側に欄がない箇所
13. 同じ責務を複数のプロンプトが異なる形で持っている箇所
14. 古い方針が残っていて、現在の思想とズレている箇所
15. 商用開発で使うには危険な曖昧表現

## レビュー観点

### 1. 全体方針の一貫性

スターターキット全体で、以下の方針が一貫しているか確認してください。

* いきなり実装しない
* 仕様、設計、フロー、テスト計画、レビューを分ける
* 各工程の成果物を次工程に引き渡す
* AIが未決事項を勝手に補完しない
* 人間が判断できる形で出力する
* 実装着手前にレビューと承認を行う

### 2. AIの権限範囲

AIに許可されている作業と、許可されていない作業が明確か確認してください。

特に以下を確認してください。

* AIが勝手に仕様を確定していないか
* AIが人間承認なしに次工程へ進める記述がないか
* 実装工程・バグ修正工程のプロンプトに `tasks.md` の直接更新指示がないか
* ドキュメント作成工程で `tasks.md` の最小更新を超えて、仕様・設計・レビュー結果の詳細を書き込ませる指示がないか
* AIが正式仕様ではない `docs/context/` の情報を仕様として扱っていないか
* AIがGO/STOP判定を人間の承認済みとして扱っていないか
* AIが「問題なさそう」などの曖昧な判断で実装に進めていないか

### 3. 成果物の接続

各工程の成果物が、次工程で明示的に参照されているか確認してください。

例：

* 仕様作成結果が設計作成で参照されているか
* 設計結果がフロー作成で参照されているか
* フロー結果がテスト計画で参照されているか
* テスト計画がレビュー観点作成で参照されているか
* レビュー結果が実装可否判断に使われているか
* バグ調査結果が修正計画に接続されているか
* 修正計画が実装プロンプトに接続されているか

### 4. テンプレートとの整合

プロンプトが要求する出力項目と、テンプレートの項目が一致しているか確認してください。

以下の不整合を検出してください。

* プロンプトでは出力を求めているが、テンプレートに欄がない
* テンプレートに欄があるが、プロンプトで記入指示がない
* 同じ意味の項目が異なる名前で存在している
* 必須項目と任意項目の扱いがズレている
* レビュー判定欄の名称や選択肢が統一されていない
* 未決事項、確認事項、制約事項の扱いが統一されていない

### 5. GO/STOP判定

レビュー工程で、人間が判断できる形の判定が出力されるか確認してください。

特に以下を確認してください。

* 総合判定があるか
* 判定値が統一されているか

  * GO
  * 条件付きGO
  * STOP
* 条件付きGOの場合、条件が明示されるか
* STOPの場合、停止理由と次に必要な作業が明示されるか
* AIが実装着手承認欄を勝手にチェックする指示がないか
* レビュー観点のチェックと、実装着手承認が混同されていないか

### 6. `tasks.md` の扱い

`tasks.md` は現在地メモとして扱い、AIの更新可否は工程種別によって分ける方針と矛盾していないか確認してください。

以下を検出してください。

* ドキュメント作成工程で、現在地メモの最小更新を超えて仕様・設計・レビュー結果の詳細を書き込ませる指示
* 実装工程・バグ修正工程で、`tasks.md` を直接更新させる指示
* `tasks.md` を正式仕様のように扱う記述
* `tasks.md` と他成果物の責務重複
* 作業報告として候補を出すべきところを、直接反映する指示になっている箇所

### 7. `docs/context/` の扱い

`docs/context/` は正式仕様ではなく、矛盾検出や判断材料のための参考情報です。

以下を確認してください。

* `docs/context/` の情報を正式仕様として扱っていないか
* 古い情報、却下案、未確定情報が混在し得る前提になっているか
* `docs/context/` を参照した場合に、正式仕様との差分や矛盾を指摘する流れになっているか
* `docs/context/` の内容をAIが勝手に採用する指示がないか
* 人間判断が必要な箇所を未決事項として明示する指示があるか

### 8. バグ修正フロー

バグ修正フローにおいて、以下が分離されているか確認してください。

* 仕様バグ
* 設計バグ
* 実装バグ
* テストバグ
* ドキュメント不整合
* 運用上の問題

また、以下を確認してください。

* バグ報告から調査へ接続されているか
* 調査結果から修正計画へ接続されているか
* 修正計画に人間承認が必要であることが明記されているか
* 仕様変更が必要な場合、実装修正だけで済ませない流れになっているか
* 仕様・設計・テスト計画の更新が必要な場合、別作業として扱われているか

### 9. 重複・責務衝突

複数のプロンプトやテンプレートが、同じ責務を異なる形で持っていないか確認してください。

以下を検出してください。

* 同じ成果物を複数プロンプトが生成する
* 同じ判定を複数箇所で異なる基準で行う
* README、AGENTS.md、個別プロンプトでルールが重複し、表現がズレている
* テンプレートとチュートリアルで説明が食い違っている
* 古いフローと新しいフローが併存している

### 10. 商用開発への適用性

スターターキットが商用開発・既存開発でも使える説明になっているか確認してください。

以下を確認してください。

* 「AIがよしなにやる」前提になっていないか
* 人間が確認・判断・承認できる形になっているか
* 既存開発フローに接続しやすい説明になっているか
* 責任の所在が曖昧になっていないか
* AIの判断結果をそのまま正として扱っていないか
* レビュー記録や判断結果が残る形になっているか

## 禁止事項

* ソースコードの修正指示に脱線しない
* 開発対象機能の仕様レビューに脱線しない
* 未決事項をAIが勝手に確定しない
* 実装工程・バグ修正工程で `tasks.md` を直接更新しない
* ドキュメント作成工程でも、`tasks.md` に仕様・設計・レビュー結果の詳細を書き込まない
* `docs/context/` の情報を正式仕様として扱わない
* README や AGENTS.md の方針を暗黙に変更しない
* 指摘だけで終わらせず、修正対象と修正方針を明示する
* 人間承認が必要な事項を、AI判断済みとして扱わない

## 実行時の注意

レビュー対象が多い場合は、一度にすべてを完璧に処理しようとせず、以下の順で確認してください。

1. `AGENTS.md` と `README.md`
2. `prompts/`
3. `docs/templates/`
4. `docs/tutorials/`
5. `docs/context/` と `tasks.md` の運用説明
6. バグ修正フロー
7. レビュー判定フロー

ただし、最終的な出力では、ファイル単位ではなく、矛盾・不整合の重要度順に整理してください。
