# docs/context/ の横断探索を行ってください

このプロンプトは、`docs/context/`（会議メモ・チャット補足・過去判断・未決事項・却下案・注意事項などの補助コンテキスト）を横断的に確認し、正式資料との関係を**候補として整理する**ための専任プロンプトです。

通常レビュー（`prompts/review_feature.md`、`prompts/review_command.md`）やバグ調査（`prompts/investigate_bug.md`、`prompts/create_bug_fix_plan.md`）は、`docs/context/` を軽い確認トリガーとしてのみ扱い、横断探索は主責務にしません。context 量が増えても通常レビューを完遂できるようにするため、深掘りが必要な場合はこのプロンプトに委譲します。

このプロンプトは**候補出し専用**です。問題や候補を見つけても、このプロンプトではファイルを変更しません。採用・却下・保留の判断は人間が行い、採用されたものだけを別作業として正式資料へ反映します。

---

## 重要な前提

- `docs/context/` は確定仕様ではありません。そのまま仕様・設計・実装方針として採用してはいけません。
- 正式資料（`20_spec.md`、`21_design.md`、`22_flow.md`、`23_test_plan.md`、`10_overview.md`、`common_design/` など）が常に基準です。
- このプロンプトが行うのは、`docs/context/` と正式資料の関係を**候補として整理し、チャットで報告する**ことだけです。
- AIは、採用・却下・保留を判断しません。候補を提示し、判断は人間に委ねます。
- 正式資料と矛盾する場合や、AIが確定可否を判断できない場合は、勝手に解決せず人間確認事項として報告します。

## review_feature_source.md / review_feature.md との違い

| | review_context.md（このプロンプト） | review_feature.md / review_command.md |
|---|---|---|
| 主対象 | `docs/context/` と正式資料の関係 | 正式資料・実装・テストの整合 |
| docs/context/ | 横断探索の主責務 | 軽い確認トリガー（深掘りはこのプロンプトへ委譲） |
| 修正 | しない（候補のみ） | しない（指摘のみ） |
| 出力先 | チャットで報告する | レビュー結果ファイルに記録する |

## 利用者が指定する項目

以下を必要に応じて指定してください。

- 対象範囲: `command/app 名、feature 名、または「全体」など。なければ docs/context/ 全体`
- 関連しそうな正式資料: `分かっていれば指定する。なければ「AIが推定」`
- 補足条件: `必要に応じて書く。なければ「なし」`

## 参照するファイル

必ず現在の以下を読み直してください。

- `AGENTS.md`
- `docs/context/README.md`
- `docs/context/` 配下の補助コンテキストメモ
- 関連しそうな正式資料（例）
  - `docs/<command_or_app_name>/10_overview.md`
  - `docs/<command_or_app_name>/features/<feature_name>/20_spec.md`
  - `docs/<command_or_app_name>/features/<feature_name>/21_design.md`
  - `docs/<command_or_app_name>/features/<feature_name>/22_flow.md`
  - `docs/<command_or_app_name>/features/<feature_name>/23_test_plan.md`
  - `docs/<command_or_app_name>/common_design/` 配下

`docs/<command_or_app_name>/bugs/` 配下は主対象ではありません。バグ報告・調査・修正計画との関係を確認する必要がある場合に限り、補助資料として参照して構いません。

## 変更してよいファイル

このプロンプトでは、いかなるファイルも変更しません。

## 変更してはいけないファイル

- `docs/context/`（補助資料として参照するだけにとどめ、変更しない）
- 正式資料（`20_spec.md`、`21_design.md`、`22_flow.md`、`23_test_plan.md`、`10_overview.md`、`11_integration_test_plan.md`、`common_design/` など）
- `docs/<command_or_app_name>/bugs/` 配下
- feature 実装ファイル、テストファイル
- `src/`、`tests/`
- `prompts/`
- `AGENTS.md`
- `README.md`
- `docs/templates/`
- 各種レビュー結果ファイル（`25_review_result.md`、`12_command_review_result.md` など）

候補を見つけても、勝手に正式資料や実装を直さないでください。候補としてチャットで報告するだけにとどめます。

## 作業手順

1. `AGENTS.md` と `docs/context/README.md` を確認してください
2. `docs/context/` 配下のメモを確認し、それぞれのメモ種別と確定状況（未決 / 保留 / 却下 / 確定前 / 参考）を把握してください
3. 関連しそうな正式資料を読み直してください
4. 各メモと正式資料の関係を、以下の確認区分で整理してください
5. チャットで候補を報告してください

---

## 確認区分

`docs/context/` の各メモを、正式資料と照らし合わせて以下のいずれか（複数該当可）に整理してください。

1. **正式資料への反映候補**：正式資料へ取り込む価値がありそうだが、まだ反映されていない内容
2. **矛盾候補**：正式資料と食い違っている、または食い違っている疑いのある内容
3. **未決事項**：まだ決まっていない、人間の判断待ちの内容
4. **却下案の混入候補**：一度見送った却下案が、実装・テスト・正式資料に混入していないかの確認ポイント
5. **人間確認事項**：AIが確定可否を判断できず、人間に判断を委ねる内容

各候補には、根拠となるメモのファイル名・確定状況・関連する正式資料・なぜその区分と判断したかを添えてください。

## 出力形式

以下の形式でチャットに報告してください。

---

### 1. 確認したファイル

参照した `docs/context/` のメモと、正式資料の一覧を記載してください。

### 2. 正式資料への反映候補

反映候補があれば、根拠メモ・関連する正式資料・反映するとどうなるかを記載してください。なければ「該当なし」と記載してください。

### 3. 矛盾候補

正式資料との矛盾候補があれば、どのメモとどの正式資料が、どう食い違っているかを記載してください。なければ「該当なし」と記載してください。

### 4. 未決事項

未決のまま残っている事項を記載してください。なければ「該当なし」と記載してください。

### 5. 却下案の混入候補

却下済みの案が実装・テスト・正式資料に混入していないか、確認すべきポイントを記載してください。なければ「該当なし」と記載してください。

### 6. 人間確認事項

AIが確定可否を判断できなかった事項を記載してください。なければ「該当なし」と記載してください。

### 7. 次に進む場合の候補

- 反映候補がある場合: 人間が採用・却下・保留を判断し、採用分だけを別作業として正式資料へ反映する
- 矛盾候補・人間確認事項がある場合: 人間が確認し、必要なら正式資料側を整える別作業を依頼する

**このプロンプトでは正式資料を変更しません。** 反映は、人間が採用対象を判断した後、別作業としてAIに依頼してください。

---

## 禁止事項

- `docs/context/` の変更
- 正式資料の変更
- `docs/<command_or_app_name>/bugs/` 配下の変更
- feature 実装ファイル・テストファイルの変更
- レビュー結果ファイル（`25_review_result.md`、`12_command_review_result.md` など）の作成・更新
- `tasks.md` の更新
- 採用・却下・保留の判断（人間に委ねる）
- `docs/context/` の内容を確定仕様として扱うこと

このプロンプトは、`docs/context/` の横断探索と候補出しだけを行います。
