# feature 本体を実装して単体テストを作成してください

あなたは、このリポジトリのAI駆動開発ルールに従う開発補助者です。

このプロンプトファイルは直接書き換えず、チャットでこのファイルのパスと下の「利用者が指定する項目」を指定して使います。
それ以外の作業ルールは変更せず、`AGENTS.md` と `docs/templates/` に従ってください。

## 利用者が指定する項目

- 対象機能フォルダ: `docs/<command_or_app_name>/features/<feature_name>/`
- コマンド/アプリ名: `<command_or_app_name>`
- 対象機能名: `<feature-name>`
- 実装ファイル: `src/<command_or_app_name>/features/<feature_name>.py`
- テストファイル: `tests/<command_or_app_name>/features/test_<feature_name>.py`
- 作りたいもの: `必要に応じて補足を書く`
- 補足条件: `必要に応じて書く。なければ「なし」`

## 実装前確認（承認ゲート）

実装を開始する前に、以下を必ず確認してください。

1. 対象機能フォルダの `24_review_checklist.md` を読んでください
2. `24_review_checklist.md` に「実装着手承認欄」セクションが存在するか確認してください
3. 「実装着手承認欄」が存在しない場合は、実装を開始せず以下のメッセージで停止してください

```
STOP — 24_review_checklist.md に実装着手承認欄がありません。
prompts/create_review_checklist.md で承認欄付きの 24_review_checklist.md を作成または更新し、人間が確認してチェックを入れるまで、feature 実装には進めません。
```

AIが承認欄を勝手に追加して実装に進んではいけません。承認欄の作成・更新は `prompts/create_review_checklist.md` を使い、人間が確認してから再実行してください。

4. 承認欄のすべてのチェックボックスにチェック（`[x]`）が入っているか確認してください
5. 未チェック項目が1つでもある場合は、実装を開始せず以下のメッセージで停止してください

```
STOP — 実装着手承認欄に未チェック項目があります。人間が確認してチェックを入れるまで、feature 実装には進めません。

未チェック項目:
- <未チェック項目を列挙>
```

AIは承認欄にチェックを入れて実装に進んではいけません。チェックを入れるのは人間です。
承認欄がすべてチェック済みであることを確認した後に、以下の作業範囲に進んでください。

## 作業範囲

1. 仕様・設計・呼び出し定義・テスト設計・レビュー観点を確認してください
2. 実装前に、以下を簡潔に提示してください。
- 参照したファイル
- 変更予定ファイル
- 実装する内容
- テストで確認する内容
- 仕様・設計・テスト計画の不足や矛盾の有無
- 要検討項目・未解決項目・人間判断待ち項目の有無
不足・矛盾・要検討項目・人間判断待ち項目がある場合は、実装を開始せず確認事項として提示してください。
要検討項目が「記録済み」であっても、「解決済み」とは扱わないでください。
3. 関数呼び出し定義に従って実装してください
4. 実装をセルフレビューしてください
5. テスト設計に従って単体テストを作成してください
6. テストをレビューしてください
7. 可能であれば `python -m pytest` を実行してください

## 共通設計書の確認

`docs/<command_or_app_name>/common_design/` が存在する場合は、実装前に以下を確認してください。

- DB設計（`33_db_design.md`）が存在する場合、DBアクセスの実装がDB設計書の定義と整合しているか
- ファイル設計（`31_file_design.md`）が存在する場合、ファイル入出力の実装がファイル設計書の定義と整合しているか
- 共通データ設計（`32_data_design.md`）が存在する場合、データ項目の実装が共通データ設計書の定義と整合しているか

共通設計書に未定義の設計が必要だと判断した場合は、実装に勝手に追加せず、作業報告に「共通設計書への追加候補」として記載してください。

## 禁止事項

- 仕様にない便利機能を追加しないでください
- 外部ライブラリを追加しないでください
- 共通化候補があっても、人間レビュー前に `src/common/` へ切り出さないでください
- entrypoint.py や entrypoint テストを作成しないでください
- entrypoint が必要な場合は、`prompts/implement_entrypoint.md` を使ってください
- 結合試験が必要な場合は、`prompts/implement_integration_test.md` を使ってください
- 結合試験、外部API、CI/CD、デプロイ資産を作成しないでください
- 設計書と大きく異なる実装に勝手に変更しないでください
- 共通設計書（DB設計・ファイル設計・共通データ設計）と矛盾するDBアクセス、ファイル入出力、データ項目を実装に追加しないでください

## 作業後の tasks.md 更新

作業後は、対象 command/app または feature の `tasks.md` を必ず確認してください。

必要に応じて、現在の状態、作業メモ、次に確認すること、引き継ぎに必要な短い注意点だけを更新してください。

`tasks.md` には、仕様・設計・テスト計画・レビュー結果の詳細や長いテストログを書かないでください。
詳細は `20_spec.md`、`21_design.md`、`22_flow.md`、`23_test_plan.md`、`24_review_checklist.md`、レビュー結果ファイルなど、それぞれの専用ファイルに記録してください。

`tasks.md` を更新しない場合は、更新しない理由を作業報告に書いてください。
